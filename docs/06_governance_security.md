# 06 · 거버넌스 · 보안 · 권한 · 감사

> "실행형" AI는 실제로 발주하고 메일을 보내고 ERP를 바꾼다. 그래서 거버넌스가 기능이 아니라 **전제**다.
> 통제 가능한 자율성(governed autonomy)이 AXOS의 신뢰 기반이다.

---

## 1. 데이터 거버넌스 — Unity Catalog 단일 통제점

- **SSOT**: 모든 데이터·AI 자산(테이블/볼륨/모델/벡터인덱스)을 UC로 일원화. 계보(lineage)·권한·감사 통합.
- **3단 네임스페이스**: `catalog.schema.table` — 예: `axos.bronze.erp_orders`, `axos.gold.demand_features`.
- **메달리온 권한**: Bronze(원본, 제한)→Silver(정규화, 도메인팀)→Gold(피처/KPI, Agent·분석).
- **민감정보**: HR/개인정보는 **컬럼 마스킹·행 필터**(UC). Agent는 마스킹된 뷰만 접근.
- **계보**: 어떤 결정이 어떤 데이터에서 나왔는지 추적 가능(evidence_ref → UC lineage).

---

## 2. 권한 모델 — 최소권한 + 역할 분리

| 주체 | 권한 원칙 |
|------|-----------|
| Agent(서비스 주체) | 도메인 Gold 읽기 + 자기 결정의 감사 쓰기만. 레거시 쓰기는 **브리지 경유만** |
| 레거시 통합계정 | 시스템별 전용 계정, 읽기/쓰기 분리, 최소권한, 한도 |
| 사람(승인자) | 역할 기반(발주 승인자/재무 승인자…). 이중승인 임계 |
| 브리지 | 유일하게 판단↔실행을 잇는 권한. 직접 호출 경로 차단 |

- **역할 분리(SoD)**: 판단을 낸 Agent와 승인하는 사람은 분리. 자동(L4)도 사후 감사 주체 분리.

---

## 3. 실행 보안 (브리지 게이트)

1. **인증**: 모든 호출 토큰 인증 — n8n `N8N_WEBHOOK_TOKEN`(현 `dev-local-token` 폴백), 브리지 토큰, 콜백 토큰.
   - ⚠️ 운영 전환 시 강한 토큰으로 교체(현 dev 폴백은 로컬 전용). → docs/01 STEP7, n8n-pipeline RESUME §8.
2. **승인 게이트(HITL)**: 되돌릴 수 없는·외부로 나가는 행동은 기본 승인(docs/04 §3).
3. **멱등성**: `decision_id` 중복 차단 → 중복 발주/중복 메일 방지.
4. **한도(Guardrail)**: 금액/수량/빈도 한도. 초과 시 자동 승급(승인 강제) 또는 거부.
5. **드라이런**: 모든 외부 어댑터 `dry_run` 지원 → 시연·검증 시 실제 영향 0.
6. **킬 스위치**: 시나리오별 즉시 중단 플래그(현장 통제감 = 수용성, docs/03).

---

## 4. 감사 (Audit) — 모든 행동의 진실 원천

- **무엇을 기록**: 누가(Agent/사람)·언제·무엇을·왜(근거 봉투)·결과.
- **저장**: live = `axos.audit.decision_log`(Delta, 불변/append). mock = `mock/audit.jsonl`.
- **용도**: ① 규제/내부감사 ② STEP10 대시보드(사용량/ROI/오류율) ③ STEP9 업무기억 적재 ④ 사고 추적.
- **레코드**: docs/04 §5 감사 스키마.

---

## 5. AI 안전 · 환각 통제

- **근거 강제**: `DecisionEnvelope.evidence` 가 비면 브리지가 거부. "근거 없는 결정 금지".
- **신뢰도 임계**: confidence<임계 → 자동 실행 불가, 사람에게.
- **출처 노출**: Copilot/Agent 모든 답에 출처(문서/데이터) — 신뢰·검증(docs/03 저항대응).
- **모델 평가**: Mosaic AI 평가로 회귀 감지. 04 evaluation-simulate가 시나리오 검증.

## 6. 컴플라이언스 · 산업별 고려 (목표 사용자: 제조/물류/유통/공공/국방/병원/교육)

| 산업 | 추가 통제 |
|------|-----------|
| 공공/국방 | 망분리·온프레/주권클라우드, 강한 감사·접근통제, 외부 LLM 제한(로컬 sLLM 옵션) |
| 병원 | 환자정보(PHI) 마스킹·접근제한, 처리 목적 제한, 강한 로깅 |
| 제조 | 제어계통(OT) 쓰기 금지(읽기 전용), 안전 우선 |
| 금융성 데이터 | 전표/발주 SoD, 이중승인, 한도 |
| 교육 | 학생 개인정보 보호 |

- **배포 선택지**: 외부 LLM 불가 환경은 로컬 sLLM(`LOCAL_SLLM_BASE_URL`, 현 mock) + 온프레 Databricks/대체 Lakehouse.

## 7. 운영 거버넌스 루틴

- **분기 레벨 재평가**: 시나리오별 L레벨을 오류율·감사 기반 재심사(상향/하향).
- **변경관리**: 계약(contracts/) 변경은 버전·리뷰 필수. 계약 불변이 레이어 독립성의 핵심.
- **사고 대응**: 잘못된 자동 실행 → 킬스위치 → 보상 트랜잭션 → 감사 → 사후분석 → 레벨 하향.
